Purpose and Scope

This Information Security Policy establishes the framework for protecting Hatch Labs Holdings LLC's information assets, systems, and data. This policy ensures the confidentiality, integrity, and availability of information and supports compliance with applicable legal, regulatory, and contractual requirements.

1. Access Control and Identity Management

1.1 User Access Management

  • Principle of Least Privilege: Users are granted minimum access necessary to perform their job functions
  • Role-Based Access Control (RBAC): Access permissions are assigned based on predefined roles and responsibilities
  • Regular Access Reviews: User access is reviewed quarterly and adjusted based on role changes or termination

Multi-Factor Authentication (MFA)

Mandatory MFA is required for:

  • All critical systems (Supabase, Google Workspace, Mercury Bank)
  • User access to Plaid Link for identity verification
  • Administrative access to any system containing user data
  • Remote access to company resources

1.2 Account Management

  • Unique user accounts for each individual
  • Strong password requirements (minimum 12 characters, complexity requirements)
  • Account lockout policies after failed login attempts
  • Immediate account deactivation upon termination or role change
  • Shared accounts are prohibited

2. Data Classification and Handling

2.1 Data Classification Levels

  • PUBLIC: Information intended for public disclosure (marketing materials, public website content)
  • INTERNAL: Information for internal business use (internal policies, business plans)
  • CONFIDENTIAL: Sensitive business information requiring protection (user data, financial information)
  • RESTRICTED: Highly sensitive information requiring the highest level of protection (authentication credentials, encryption keys, sensitive personal data, financial account data)

Personal Data Protection

Given AmandasList's collection of personal and financial data through Plaid:

  • All personal data classified as Confidential or Restricted
  • Financial account data classified as Restricted
  • Identity verification data classified as Restricted
  • Special handling procedures for financial data per applicable regulations

3. Encryption Standards

3.1 Data Encryption at Rest

  • Primary Database: AES-256 encryption via Supabase hosted PostgreSQL
  • Minimum Standard: AES-256 encryption for all databases containing personal or financial data
  • Key Management: Encryption keys managed by Supabase with industry-standard key rotation
  • Scope: All user data, financial data from Plaid, identity verification data, and business-critical information

3.2 Data Encryption in Transit

  • Minimum Standard: TLS 1.2 or higher for all data in transit
  • Implementation: TLS 1.3 where supported, with perfect forward secrecy
  • API Communications: All API calls to third-party services (Plaid, Supabase) use TLS 1.2+
  • Web Application: HTTPS enforced with HSTS headers
  • Certificate Management: Automated certificate renewal via Cloudflare

4. Incident Response Procedures

4.1 Incident Response Team

  • Incident Commander: CEO (Amanda Bradford)
  • Technical Lead: Development team lead (currently CEO)
  • External Resources: Cybersecurity consultant (as needed)
  • Legal Counsel: External legal counsel (as needed)

4.2 Response Procedures

  1. Detection and Analysis (0-1 hour): Identify and verify the incident, assess scope and potential impact
  2. Containment (1-4 hours): Immediate containment of the incident, prevent further damage
  3. Eradication and Recovery (4-24 hours): Remove threat and vulnerabilities, restore systems
  4. Post-Incident Activities (24-72 hours): Lessons learned review, update policies, prepare incident report

Communication and Notification

  • Internal notification within 2 hours of confirmed incident
  • Customer notification per applicable regulations (CCPA, etc.)
  • Regulatory notification as required by law
  • Vendor notification for third-party service impacts

5. Vulnerability Management

5.1 Vulnerability Assessment

  • Automated dependency scanning via GitHub Dependabot
  • Regular security assessments of critical systems
  • Third-party security testing annually or after major changes
  • Continuous monitoring for new vulnerabilities

5.2 Patch Management

  • Critical security patches applied within 72 hours
  • High-priority patches applied within 7 days
  • Regular patches applied within 30 days
  • Emergency patching procedures for zero-day vulnerabilities

7. Business Continuity and Disaster Recovery

7.1 Backup and Recovery

  • Database Backups: Automated daily backups via Supabase with point-in-time recovery
  • Application Code: Version control and automated deployment pipelines
  • Configuration Data: Infrastructure as code with version control
  • Recovery Testing: Quarterly backup restoration testing

7.2 Business Impact Analysis

  • Critical business functions identified and prioritized
  • Recovery time objectives (RTO) and recovery point objectives (RPO) defined
  • Dependencies on third-party services documented
  • Alternative processing capabilities identified

8. Compliance and Legal Requirements

8.1 Regulatory Compliance

  • California Consumer Privacy Act (CCPA): Privacy rights for California residents
  • State Privacy Laws: Compliance with applicable state privacy regulations
  • Financial Data Protection: Compliance with financial data protection requirements

8.2 Audit and Assessment

  • Annual security assessment by qualified third party
  • Internal security reviews quarterly
  • Compliance audits as required by regulations
  • Documentation retention for audit purposes

9. Employee Security Requirements

9.1 Security Awareness Training

9.2 Acceptable Use
